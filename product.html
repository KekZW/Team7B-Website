Design Process & Product Overview

Understanding the Problem

Problem Statement

"When conducting activities by the river, the condition of the river greatly impacts safety aspects. The Scottish Environment Protection Agency (SEPA) does publish data around river flow, but it is difficult to decipherâ€”especially when it comes to predicting river conditions ahead of time. Another aspect is that the river flow rate upstream affects pollution levels downstream, but again, this is difficult to derive from the available data."

Partner Interview & Clarifications

During our initial meeting with our project partner Ms Sally fron Friends of the River Kelvin (FORK), we were able to probe her for more specifics about the project

  • When talking about river conditions, what is it about?
    We are talking about river flow rate and water level.
  • Who would be using the data?
    The data would be used by the member of the public as well as volunteers.
  • What kind of end product are you looking for?
    We are open to any ideas, just that it should be easy to maintain and easy to use.

What we understood

Our partner also provided additional information, from which we obtained the following insights:

  • There is a need to predict river conditions ahead of time.
  • SEPA provides data, but it is difficult to decipher, making it harder to use for predictions
  • We also need to account for pollution levels.
Refining the problem statement

We need a system that is easy to maintain and easy to use, to help us predict and display current river conditions (Water Level + River Flow Rate). The app can be used by both the public and volunteers. We are open to any ideas, but the end product should be easy to maintain and easy to use. We also need to be able to show pollution levels in an easy-to-digest way to educate members of the public.

Brainstorming Ideas

Ideation Sketch

Our team collaboratively explored multiple solution concepts by initially sketching ideas on paper. This hands-on brainstorming approach helped us visualize and iterate through several possibilities, including mobile applications, websites and an information display system. After further discussion, we were able to wither down our range of feasible ideas and refinethe more promising directions for further development.

We also looked to assess the technical feasibility of each concept, evaluating factors like data integration, user experience and scalability. Through collaborative discussions and team voting, we decided to proceed with a mobile application as our proposed solution.

River Information Display System

Install digital Information Display System, similar to those at the beaches or bus stop that display real-time information about the rivers.

QR Code + Website

Implement a QR code system that links to a website providing real-time river data and alerts.

Mobile Application

A mobile app that provides real-time updates on river conditions, safety alerts and personalized activity suggestions.

Research

To evaluate the feasibility of our idea, we explored potential data sources and investigated how to extract relevant information from them.

Water Data

We found that the Scottish Environment Protection Agency (SEPA) provides various water-related datasets, accessible through their API. Examples include:

  • Water Level (Dryfield) View API Endpoint
  • Water Flow (Bridgend) View API Endpoint
  • Water Level & Quality (Killermont) View API Endpoint
  • Rainfall Levels View API Endpoint

Pollution Data

SEPA also provides pollution data; however, it is not available via API. Instead, it must be manually downloaded from their Water Classification Hub. Users can search for a river, select it, and download the CSV data.

Weather Data

Several APIs are available for obtaining weather information: Water Classification Hub. Users can search for a river, select it, and download the CSV data.

  • WeatherAPI
    Offers a free tier, but with limited API calls and some features locked behind paid plans. One limitation is the inability to select specific locations like the River Kelvin â€” only broader locations such as Glasgow are available.
  • OpenWeather API
    Offers a free tier with limited API calls, but cannot be used for commercial purposes.

Prototype Refinement and Feedback

Initial High-Fidelity Protytype

After the initial draft of the low-fidelity wireframes, we began testing and found several areas for improvement. The first high-fidelity design focused on enhancing the user interface, improving accessibility and integrating user feedback.

Second Partner Meeting
Partner discussion

With the second partner meeting with Ms Sally, we were able to showcase our high-fidelity prototypes and solicit additional feedback.

  • What do you think about this prototype?
    It was good, but we should consider why people would want to use this application and how we should advertise it.
  • Is there any features we should consider?
    Maybe a few more features, like news updates and community links to the FORK.

Main Insights

From our second partner meeting, we took away the following insights:

  • Login System - We force users to create an account to use the app, which would allow us to gather more data and also allow users to save their preferences.
  • Guest System - We allow users to use the app without creating an account, which would allow for a more seamless experience but would limit our ability to gather data and save preferences.
Additional Discussion

Our partner also questioned us about our research on the data sources. We explained that while the official Scottish Environmental Protection Agency (SEPA) made certain data predictable, there were issues with the relevance of the data (outdated by 2 years) We further discussed the limitations of using (other) available public resources to perform forecasting. Ms Sally considered this limitation and asked us place our suggestions in the poster instead of the prototype (we decided to place our suggestions under the 'future work' section).

Integrative Thinking

Integrative thinking played a crucial role in our design process. By synthesizing diverse perspectives and ideas, we were able to create a more holistic solution. This approach encouraged collaboration and open dialogue within the team, leading to innovative design outcomes.

During this stage, we had a few ideas and also concerns regarding the login system. We considered two different approaches:

Login System:

Require users to create an account to use the app. This would allow us to gather more data and enable users to save their preferences.

Guest System:

Allow users to use the app without creating an account. This would create a more seamless experience but would limit our ability to gather data and save preferences.

Usability Testing

Overview

At this stage, we refined our design with internalÂ testing. We compared different options, like when to require login (at launch or only for reporting), and how to display pollutants (table vs. icons). These tests helped us understand user preferences and usability better. We also conducted internal reviews, where the team voted on the final layout. This collaborative approach ensured that the chosen design reflected both user input and team consensus.

Feedback
Issues
  • A help option would be nice to help user understand what is going on
  • Have more news about what is going on (Friends of the River Kelvin)
  • Change the QR code, as there is no way for users to scan it
Solutions
  • Add a help button at the top right of the screen for users to access more information.
  • Add a scrollable section in the home page that has the latest updates
  • In the actual app, allow the QR code will be clickable and link to their respective sites

Final High Fidelity Design

Based on feedback, we refined the final design to include improvements such as enhanced accessibility, clearer icons and redesigned UI elements.

Our final product is a mobile app called River Kelvin Watch. It enables users to:

  • Check real-time river flow and water level
  • Check forecasted river flow and water level
  • Understand data through simplified charts and icons
  • View historical patterns for awareness and planning
  • Report river condition via intuitive user interface

Future work

Even though we are only tasked with creating a prototype, we have some ideas for future work that could enhance the app:

  • Test and integrate real time data (API) into the app
  • Test and integrate the forecasting function, using both weather forecast and historical data
  • Integrate "What3Words" for map view and activities
  • Implement a new way to automatically get latest news from the FORK website
  • Explore more ways to present the pollution data and combine results from River Fly Monitoring and the offical SEPA data
  • Add a new UI element that shows the current/forecast weather condition
